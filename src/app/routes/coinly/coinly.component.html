Coinly

http://gocoinly.herokuapp.com

Collection page where user’s can view every U.S. coin and add them to their collection.

Inspiration

When I was younger I used to collect coins. I spent hours organizing and documenting my inventory. It was always inconvenient to have to pull out my entire collection or to look through physical pages of Excel printouts to figure out if I already owned a particular coin. I wondered if there might be any programs that could track my collection electronically. There were options, but they were all rudimentary at best and cost a fair amount of money.

In early 2019 and I decided to search again to see if the tools had evolved at all. I was surprised that the same exact desktop applications were the ones that popped up in the search results. All with their early 2000’s (if not older) looking user interfaces and all still desktop based applications. I saw this as the perfect opportunity to take two things that I enjoy and using them to complement each other as well as give me the opportunity to learn something new.

The Stack

Starting this project out, I had the goal of learning how to use some new technologies. I decided to try learning React.js since it had been a framework that I had been wanting to try out for a long time. I had prior experience working with LESS as a CSS preprocessor, but in a corporate setting, so I wanted to dive a little deeper into the technology and develop my own design system.

I spent much of my senior year coding in Python and decided that this could also be a good opportunity to play around with a new server side technology. I had only ever used Node.JS prior to this project. I started the project out using SQLite as my database, but when it came time to deploying the application, I found that it would be easier to maintain data if it were hosted somewhere where I had more control over the information.

Development

I began this project by researching how to setup a basic React.js project. After running a couple commands that I found on Facebook’s React.js tutorial page. The next step I took was to make sure the application would build properly and ensure that I had the project setup the way that I wanted it. The default React application that is generated did not allow me to utilize LESS. I quickly found resources that helped me install and configure the LESS compiler in my project.

Once the frontend part of the project was setup to my liking, I built a quick Flask application in Python to serve the initial page to me so that I could start developing some components. Flask proved to be a quick and easy framework to implement server functionality and I would definitely recommend it for anyone that wants to create any sort of basic server to support their frontend work.

I started building out the application by creating some basic components such as buttons and inputs. I am used to component based design systems and am a strong proponent. Not only did creating these components allow me to build the application quicker and keep components consistent, but it also allowed me to learn the fundamentals of React.js and any other modern frontend framework.

Being a recent college graduate, I had very little experience with frontend framewroks. The JavaScript knowledge that I had gained up to that point had been primarily self taught and avoided higher level concepts such as data binding. Once I had built out a couple components, I figured that it was time for me to start building some of the application’s core functionality.

Displaying Data

The collections functionality took the most amount of time for me to build. The main purpose of this view is to allow users to narrow down the search results to get to a coin that they want to add to their collection. I created a JSON file that contained data about nearly every single coin that has ever been minted for circulation in the United States. The final file is just over 15,000 lines long. I use this JSON file as the model for the application. In the collections view, there are three layers. The first sorts the coins by denomination, the second by category, and the third by year and mint mark.

Once a user has navigated down to the bottom level, they are then able to add a coin to their collection by clicking the add icon in the top right of a tile. In the top left of the tiles on the second and third levels, the user will be presented one of three colors (green, yellow, or red) which signify that the user has the coin/set of coins, has partially completed a set of coins, or does not own the coin or any coin from the set of coins respectively.

Creating the User

These features would not work so well without adding in some backend data management and routing. The next task to tackle would be allowing users to create accounts. For this I first developed a schema for storing user information and then created a simple interface to collect a users information and added in some of the routing to connect the client to the database. I then added some encryption of the passwords to increase security and added sessions to maintain user state in the application.

In creating the user, I felt that it would also be good to have a place where one could go to look at their collection without all of the extra visual data that is displayed on the collections page. The inventory page has a similar setup, but only lists the coins that a user has in their collection.

Conclusions

Since this was my first experience with any current JavaScript framework, I found it difficult to wrap my head around how the application is built and renders. I found myself stuck on simple issues for hours when I first started, but by the end of this fairly limited exercise, I found myself able to develop components and views fluidly. There are features that I would eventually like to add such as a collection overview and eventually some sort of social aspects so that people can share their new finds or possibly even offer them for sale. Before spending any more time on functionality though, it would probably be in my best interest to talk to other collectors and see what suits their needs.

My experience with the new frameworks couldn’t have been any better. I love React. Once you get the hang of it, building user interfaces is incredibly simple and ensuring that your design system standards are upheld couldn’t be easier. Being able to fix an issue in one place and impact every instance that it is used in really cuts down on the time it takes to add new features and squash annoying bugs. I am hoping that I will be able to employ this framework again in future projects.
